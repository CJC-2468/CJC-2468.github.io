golang map底层实现？

底层是一个散列表，因此实现map的过程实际上就是实现散表的过程，go中实现采用拉链法解决冲突。

这个散列表里有两个结构体，一个是hmap（(a header for a go map)），一个是bmap（(a bucket for a Go map，通常叫其 bucket)）。

hmap中存有buckets数组的地址，数组中每一个元素都是一个bucket，bucket是一个链表。

我们根据key计算出一个hash，取低位找到我们要找的kv在数组的哪一个bucket（也就是在buckets数组的哪一个位置），再取高位哈希，定位到他在bucket的哪个key，这里其实是按照链表那样，如果这个bucket没找到就一个个往后面的overflow bucket找，然后取到我们要的kv。

（如果在 bucket 中没找到，并且 overflow 不为空，还要继续去 overflow bucket 中寻找，直到找到或是所有的 key 槽位都找遍了，包括所有的 overflow bucket。）

bmap 就是我们常说的“桶”，桶里面会最多装 8 个 key，这些 key 之所以会落入同一个桶，是因为它们经过哈希计算后，哈希结果是“一类”的。在桶内，又会根据 key 计算出来的 hash 值的高 8 位来决定 key 到底落入桶内的哪个位置（一个桶内最多有8个位置）。

go map如何扩容？

go语言默认负载因子阈值是6.5，超过就会双倍扩容。负载因子=元素个数/桶的个数。如果负载因子没有超过，但是溢出桶太多，那么也会扩容，但是会发生等量扩容。

翻倍扩容：容量扩容为原来的两倍，原有的 key 并不会一次性搬迁完毕，因为会造成系统卡顿，所以他是通过后面每次访问的时候顺便搬迁，每次最多只会搬迁 2 个 bucket，这种一点点搬迁的方式叫渐进式扩容。

等量扩容：重新排列。新申请同样容量大小的桶，然后一个个把旧桶的键值迁移到新桶。这种情况会发生在旧桶中很多kv被删除的情况，迁移实际上就是旧的bucket[0]迁移到新的bucket[0]，但是这些kv会紧密排在桶里，这样可以有效减少溢出桶的数量。扩容完成后，其实链表长度就变短，key 都集中到了一个 bucket，更为紧凑，提高了查找的效率。

map为什么不是并发安全的？

因为多个协程并发的向map中的某个key执行写操作的时候，最开始会出现脏数据，后来干脆编译器直接报panic:concurrent map writes的错，所以说map不是并发安全的。

但是后来go引入了效率较高的并发安全的 sync.Map，sync.Map 有以下特性：

  • 无须初始化，直接声明即可。
  • sync.Map 不能使用 map 的方式进行取值和设置等操作，而是使用 sync.Map 的方法进行调用，Store 表示存储，Load 表示获取，Delete 表示删除。

实现思想：sync.map 的整体思想用尽量将读写分离，来减少锁对性能的影响。

适用场景：两种场景都比较苛刻，要么是一写多读，要么是各个协程操作的 key 集合没有交集（或者交集很少）。所以官方建议先对自己的场景做性能测评，如果确实能显著提高性能，再使用 sync.map。

另外，还可以自己用的时候配合读写锁。
